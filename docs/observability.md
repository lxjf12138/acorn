# Observability

Acorn uses OpenTelemetry as the standard observability substrate. Kratos
middleware handles transport-level tracing and context propagation. servicekit
owns shared initialization and middleware wiring.

## Layers

```text
Kratos transport middleware
  HTTP/gRPC spans and trace propagation

servicekit observability
  OpenTelemetry provider initialization
  shared resource attributes
  server/client middleware options

Acorn domain instrumentation
  workspace/resource/sandbox spans and metrics

Acorn Event Surface
  product timeline and audit events linked to trace/span ids
```

This foundation only enables transport-level tracing. Domain spans, metrics,
audit logs, and Event Surface records are separate follow-up work.

## Resource Attributes

Every service should report:

```text
service.name
service.version
deployment.environment
acorn.service.id
```

`service.name` is the runtime service name. `acorn.service.id` is the logical
Acorn service id used in ResourceRef authority and workspace host references.

## Exporters

Tracing supports:

```text
noop    no SDK exporter
stdout  local development traces
otlp    OTLP gRPC exporter, insecure endpoint for the first version
```

Local development can use:

```yaml
observability:
  enabled: true
  environment: dev
  tracing:
    enabled: true
    exporter: stdout
    sample_ratio: 1.0
```

Production-like deployments should set `exporter: otlp` and provide an
`otlp_endpoint`.

## Span Names

Domain instrumentation should use stable, low-cardinality names:

```text
acorn.resource.upload
acorn.resource.download
acorn.resource.import
acorn.resource.export
acorn.resource.gateway.open
acorn.workspace.view.list
acorn.workspace.view.preview
acorn.workspace.transfer.import
acorn.workspace.transfer.export
acorn.workspace.lease.acquire
acorn.workspace.exec
acorn.sandbox.profile.resolve
```

Transport spans are generated by Kratos tracing middleware and should not be
renamed by domain code.

## Implemented Domain Spans

The current domain tracing layer adds spans for the main resource and workspace
boundaries:

```text
Resource
  acorn.resource.upload
  acorn.resource.download
  acorn.resource.import
  acorn.resource.export
  acorn.resource.gateway.open

Workspace
  acorn.workspace.view.list
  acorn.workspace.view.preview
  acorn.workspace.transfer.import
  acorn.workspace.transfer.export
  acorn.workspace.lease.acquire
```

These spans use the active OpenTelemetry context created by transport
middleware. They do not add metrics, audit logs, or Acorn Event Surface records.

Current span attributes are intentionally small:

```text
acorn.operation
acorn.status
acorn.sandbox.profile_id / acorn.workspace.profile_id
acorn.resource.authority_service_id
acorn.resource.mime_type
acorn.resource.size_bytes
acorn.workspace.lease.mode
acorn.workspace.lease.reason
acorn.truncated
```

## Sensitive Data

Do not record these in span attributes, events, or metric labels:

```text
environment variable values
stdout/stderr content
file content
full command args
secrets or tokens
large request/response bodies
```

When command or file details are useful, record bounded summaries or explicit
safe fields only.

## Metrics Labels

Metric labels must stay low-cardinality. Do not use:

```text
workspace_id
resource_id
user_id
session_id
file path
command args
content hash
```

Prefer stable labels such as service name, operation, status code, profile id,
backend kind, and exporter kind.

## Event Surface Boundary

OpenTelemetry traces explain system execution. Acorn Event Surface records will
explain product and audit history. Events may reference trace/span ids, but
traces are not a substitute for audit records or user-visible timelines.
